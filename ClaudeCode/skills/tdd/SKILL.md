# t-wada式TDDチェック

テストコードをt-wada式TDDの原則に基づいて分析し、品質レポートを出力する。
言語を問わず対応可能であり、9項目のチェックルールは言語非依存で記述されている。

## 実行チェックリスト

以下のチェックリストをコピーして進捗を追跡する:

```text
TDDチェック進捗:
- [ ] Step 1: テストファイル特定・言語検出
- [ ] Step 2: リファレンス読み込み
- [ ] Step 3: テストコード読み取り
- [ ] Step 4: プロダクションコード特定・読み取り
- [ ] Step 5: 9項目チェック実行
- [ ] Step 6: セルフバリデーション
- [ ] Step 7: レポート出力
- [ ] Step 8: 完了
```

## 実行手順

### Step 1: テストファイルの特定と言語検出

引数 `$ARGUMENTS` が指定されている場合はそのファイルをチェック対象とする。
指定がない場合は、以下の二段階の検出アプローチでテストファイルを探索する（両段階の結果を統合する）。

**第1段階: 主要言語のパターンマッチ**

Globツールで以下のパターンを並列検索する:

| 言語 | パターン |
|---|---|
| Python | `tests/**/*.py`, `**/test_*.py`, `**/*_test.py`, `**/*_spec.py` |
| TS/JS | `**/*.test.ts`, `**/*.spec.ts`, `**/*.test.tsx`, `**/*.spec.tsx`, `**/*.test.js`, `**/*.spec.js`, `**/*.test.jsx`, `**/*.spec.jsx`, `**/__tests__/**/*.[jt]s?(x)` |
| Go | `**/*_test.go` |

**第2段階: 汎用検出（第1段階でカバーされない言語やテストが疑われる場合）**

1. プロジェクト設定ファイル（`Cargo.toml`, `pom.xml`, `build.gradle`, `*.csproj`, `Gemfile` 等）から言語を推定する
2. `test/`, `tests/`, `spec/`, `__tests__/`, `src/test/` ディレクトリを探索する
3. ファイル名に `test`, `spec` を含むファイルを検出する
4. 検出したファイルの内容からテストフレームワークを推測する

検出された言語に対応するリファレンスがある場合は言語固有の観点を適用する。ない場合でも9項目の汎用チェックは適用可能である（t-wadaの原則は言語非依存）。

### Step 2: リファレンスの読み込み

チェックの前に、以下のリファレンスファイルを読み込んで判定基準を確認する:

- `references/t-wada-principles.md` — 必ず読み込む
- `references/language-patterns.md` — 検出された言語に応じて該当セクションのみ参照

※ スキルの `references/` ディレクトリからの相対パスで参照すること。

### Step 3: テストコードの読み取り

特定した全テストファイルの内容をReadツールで読み取る。
ファイル数が多い場合は並列で読み取ること。

### Step 4: プロダクションコードの特定と読み取り

テストファイルのimport文やパッケージ構成を解析し、テスト対象のプロダクションコードを特定する。

1. **テスト対象の特定**: import文・from文・パッケージ構成等からプロダクションコードのモジュール/ファイルを抽出する
2. **プロダクションコードの読み取り**: 特定したファイルをReadツールで読み取る（複数ファイルは並列で読み取る）
3. **テスト可能な関数/クラスの列挙**: プロダクションコードからpublicな関数・クラス・メソッドを列挙する
4. **テストとの照合準備**: 各関数/クラス/メソッドがどのテストでカバーされているかを対応付ける

この情報はチェック9（テストケースの十分性）で使用する。

### Step 5: 9項目チェックの実行

以下の9項目を順番にチェックする。各項目について **判定**、**該当箇所**、**改善提案** を記録する。
チェック項目は言語非依存の汎用ルールとして記述されており、未知の言語でも適用可能である。
主要言語（Python, TS/JS, React, Go）の場合は言語固有の補足観点も適用する。

判定は3段階: ✅ 準拠 / ⚠️ 要改善 / ❌ 違反

#### チェック1: AAAパターン

テストがArrange（準備）- Act（実行）- Assert（検証）の3フェーズで構成されているか確認する。

- Arrange: テストデータやオブジェクトの準備
- Act: テスト対象の操作を1つだけ実行
- Assert: 結果の検証
- フェーズが混在していないか
- Actが複数含まれていないか

#### チェック2: One assertion per test

各テストのアサーションが「意味のあるひとまとまり」になっているか確認する。

- 1テストに大量の無関係なアサーションが詰め込まれていないか
- 関連するアサーションのグループは許容する（例: オブジェクトの複数プロパティの検証）
- パラメータ化テストで適切に分離されているか

#### チェック3: アサーション情報量

失敗時に期待値と実際値が分かるアサーションになっているか確認する。

- 期待値と実際値が明確なアサーションを使用しているか
- 情報量の少ないアサーション（真偽値のみの検証等）はないか
- エラー/例外検証時にメッセージ内容まで検証しているか
- カスタムメッセージが必要な箇所で付与されているか

言語固有: Python `assert x`回避 / TS `toBeTruthy()`乱用回避 / Go `t.Fail()`のみ回避

#### チェック4: 自作自演の回避

テストとプロダクションコードが同じロジックで期待値を計算していないか確認する。

- テスト内で期待値をハードコーディング（リテラル値）しているか
- テスト対象と同じ計算ロジックを使って期待値を算出していないか
- 三角測量（Triangulation）のアプローチが適切に使われているか

#### チェック5: テストのドキュメント性

テスト名や説明から意図が読み取れるか確認する。

- テスト関数/メソッド名が振る舞いを記述しているか
- `test_1`, `test_foo` のような意味のない名前はないか
- テストグループの説明で文脈が説明されているか
- 失敗時にどの仕様が壊れたか分かるか

#### チェック6: モックの最小化

過剰なモックで脆いテストになっていないか確認する。

- モックは外部依存（I/O、ネットワーク、時刻等）に限定されているか
- 内部実装の詳細をモックしていないか
- モックの設定が複雑すぎないか
- テスト対象のインターフェースではなく実装に依存していないか

言語固有: React `getByRole`>`getByTestId`優先順位 / Go インターフェース経由の依存性注入

#### チェック7: テスト信頼性（偽陽性・偽陰性の回避）

テスト結果が常に真実を反映する信頼性の高い状態か確認する。

**偽陰性の検出観点**:
- アサーション文やアサーション関数が1つも無いテストはないか
- 例外検証のみでメッセージ検証がないテストはないか
- 空のテスト（`pass` 等）はないか
- 例外を握りつぶしていないか

**偽陽性の検出観点（汎用）**:
- 実装の内部詳細に依存する脆いテストはないか
- 非決定的要素（タイムスタンプ、乱数、実行順序）に依存していないか
- リファクタリング耐性が欠如していないか（パブリックAPIを通じてテストしているか）

言語固有: React `getByTestId`過剰使用・内部state依存 / Go `t.Parallel()`変数キャプチャ漏れ / TS snapshot過剰使用

#### チェック8: テストサイズ

Small（単体）テスト中心のピラミッド型になっているか確認する。

- テストの大部分がSmall（外部依存なし、高速）か
- Medium（DB、ファイルシステム等のローカルリソース依存）の割合は適切か
- Large（ネットワーク、外部サービス依存）は最小限か
- テスト実行時間に問題はないか

#### チェック9: テストケースの十分性

Step 4で収集したプロダクションコード情報を基に、テストケースが十分かを確認する。

- テスト対象のプロダクションコードからpublicな関数・クラス・メソッドを列挙する
- 各関数/クラス/メソッドに対応するテストが存在するか照合する
- テスト可能性による分類と優先度付けを行う:
  - 純粋関数（最優先）> ユーティリティ（高）> ビジネスロジック（高）> I/O依存（中）> エントリポイント（低）
- テスト済みの関数について、主要なコードパス（正常系・異常系・分岐）がカバーされているか確認する
- 不足しているテストケースを優先度付きで報告する

### Step 6: セルフバリデーション

レポート出力前に以下を確認する:

1. 全9項目にチェック結果が記録されているか
2. 各チェックで該当箇所に行番号が含まれているか
3. チェック7で偽陽性・偽陰性の両観点が検討されているか
4. チェック9でプロダクションコードのpublicな全関数・クラス・メソッドが照合されているか
5. 不足があればStep 5に戻って補完する

### Step 7: レポート出力

以下の形式でレポートを出力する:

#### サマリーテーブル

```markdown
## t-wada式TDDチェック結果

| # | チェック項目 | 判定 | 概要 |
|---|------------|------|------|
| 1 | AAAパターン | ✅/⚠️/❌ | 一行サマリー |
| 2 | One assertion per test | ✅/⚠️/❌ | 一行サマリー |
| 3 | アサーション情報量 | ✅/⚠️/❌ | 一行サマリー |
| 4 | 自作自演の回避 | ✅/⚠️/❌ | 一行サマリー |
| 5 | テストのドキュメント性 | ✅/⚠️/❌ | 一行サマリー |
| 6 | モックの最小化 | ✅/⚠️/❌ | 一行サマリー |
| 7 | テスト信頼性（偽陽性・偽陰性） | ✅/⚠️/❌ | 一行サマリー |
| 8 | テストサイズ | ✅/⚠️/❌ | 一行サマリー |
| 9 | テストケースの十分性 | ✅/⚠️/❌ | 一行サマリー |

**総合スコア: X/9 準拠**
```

#### 詳細セクション

各チェック項目について:

```markdown
### チェックN: 項目名 [判定]

**分析結果:**
具体的な分析内容

**該当箇所:**
- `ファイルパス:行番号` - 説明

**改善提案:**
- 具体的な改善案（ある場合）
```

#### 総括

全体的な評価と優先度の高い改善点を3つ以内で提示する。

### Step 8: レポート出力後の動作指針

1. **デフォルト: レポート出力のみで完了する。** 改善の実装は行わない。ユーザーがレポートを確認し、次のアクションを判断するのを待つ
2. **実装指示があった場合**: `superpowers:test-driven-development` スキルと連携し、Red-Green-Refactorサイクルに従って改善を実施する
3. **追加分析の要求**: 特定のチェック項目について深掘り分析を追加で出力する
