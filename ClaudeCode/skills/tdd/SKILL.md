# t-wada式TDDチェック

テストコードをt-wada式TDDの原則に基づいて分析し、品質レポートを出力する。

## 実行手順

### Step 1: テストファイルの特定

引数 `$ARGUMENTS` が指定されている場合はそのファイルをチェック対象とする。
指定がない場合は、プロジェクト全体から以下のパターンでテストファイルを探索する:

- `tests/`ディレクトリ配下
- `**/test_*.py`
- `**/*_test.py`
- `**/*_spec.py`

Globツールを使って対象ファイルを列挙すること。

### Step 2: リファレンスの読み込み

チェックの前に、以下のリファレンスファイルを読み込んで判定基準を確認する:

```
C:\Users\shun\.claude\skills\tdd\references\t-wada-principles.md
```

### Step 3: テストコードの読み取り

特定した全テストファイルの内容をReadツールで読み取る。
ファイル数が多い場合は並列で読み取ること。

### Step 3.5: プロダクションコードの特定と読み取り

テストファイルのimport文を解析し、テスト対象のプロダクションコードを特定する。

1. **テスト対象の特定**: import文やfrom文からプロダクションコードのモジュール/ファイルを抽出する
2. **プロダクションコードの読み取り**: 特定したファイルをReadツールで読み取る（複数ファイルは並列で読み取る）
3. **テスト可能な関数/クラスの列挙**: プロダクションコードからpublicな関数・クラス・メソッドを列挙する
4. **テストとの照合準備**: 各関数/クラスがどのテストでカバーされているかを対応付ける

この情報はチェック9（テストケースの十分性）で使用する。

### Step 4: 9項目チェックの実行

以下の9項目を順番にチェックする。各項目について **判定**、**該当箇所**、**改善提案** を記録する。

判定は3段階: ✅ 準拠 / ⚠️ 要改善 / ❌ 違反

#### チェック1: AAAパターン

テストがArrange（準備）- Act（実行）- Assert（検証）の3フェーズで構成されているか確認する。

- Arrange: テストデータやオブジェクトの準備
- Act: テスト対象の操作を1つだけ実行
- Assert: 結果の検証
- フェーズが混在していないか
- Actが複数含まれていないか

#### チェック2: One assertion per test

各テストのアサーションが「意味のあるひとまとまり」になっているか確認する。

- 1テストに大量の無関係なアサーションが詰め込まれていないか
- 関連するアサーションのグループは許容する（例: オブジェクトの複数プロパティの検証）
- パラメータ化テストで適切に分離されているか

#### チェック3: アサーション情報量

失敗時に期待値と実際値が分かるアサーションになっているか確認する。

- `assert x == expected` のように期待値と実際値が明確か
- `assert result` のような情報量の少ないアサーションはないか
- `pytest.raises` でエラーメッセージまで検証しているか
- カスタムメッセージが必要な箇所で付与されているか

#### チェック4: 自作自演の回避

テストとプロダクションコードが同じロジックで期待値を計算していないか確認する。

- テスト内で期待値をハードコーディング（リテラル値）しているか
- テスト対象と同じ計算ロジックを使って期待値を算出していないか
- 三角測量（Triangulation）のアプローチが適切に使われているか

#### チェック5: テストのドキュメント性

テスト名やdocstringから意図が読み取れるか確認する。

- テストメソッド名が振る舞いを記述しているか
- `test_1`, `test_foo` のような意味のない名前はないか
- テストクラスのdocstringで文脈が説明されているか
- 失敗時にどの仕様が壊れたか分かるか

#### チェック6: モックの最小化

過剰なモックで脆いテストになっていないか確認する。

- モックは外部依存（I/O、ネットワーク、時刻等）に限定されているか
- 内部実装の詳細をモックしていないか
- モックの設定が複雑すぎないか
- テスト対象のインターフェースではなく実装に依存していないか

#### チェック7: 偽陰性の回避

アサーションゼロのテスト（常にパスするテスト）が存在しないか確認する。

- `assert` 文が1つも無いテストはないか
- `pytest.raises` のみでメッセージ検証がないテストはないか
- `pass` だけのテストメソッドはないか
- 例外を握りつぶしていないか

#### チェック8: テストサイズ

Small（単体）テスト中心のピラミッド型になっているか確認する。

- テストの大部分がSmall（外部依存なし、高速）か
- Medium（DB、ファイルシステム等のローカルリソース依存）の割合は適切か
- Large（ネットワーク、外部サービス依存）は最小限か
- テスト実行時間に問題はないか

#### チェック9: テストケースの十分性

Step 3.5で収集したプロダクションコード情報を基に、テストケースが十分かを確認する。

- テスト対象のプロダクションコードからpublicな関数・クラス・メソッドを列挙する
- 各関数/クラスに対応するテストが存在するか照合する
- テスト可能性による分類と優先度付けを行う:
  - 純粋関数（最優先）> ユーティリティ（高）> ビジネスロジック（高）> I/O依存（中）> エントリポイント（低）
- テスト済みの関数について、主要なコードパス（正常系・異常系・分岐）がカバーされているか確認する
- 不足しているテストケースを優先度付きで報告する
