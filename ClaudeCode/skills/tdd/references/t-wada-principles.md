## テスト信頼性の本質

t-wada氏の2024年講演における核心メッセージを上位概念として示す。全ての原則はこの本質に基づく。

### 自動テストの真の目的

自動テストの目的は「信頼性の高い実行結果に短い時間で到達する状態を保つこと」である。テストスイートが信頼できなければ、開発者はテスト結果を無視し始め、テストの存在意義が失われる。

### テストの「嘘」を排除する

テストの「嘘」には2種類ある:

- **偽陰性 (False Negative)**: コードが壊れているのにテストがパスする（見逃し）
- **偽陽性 (False Positive)**: コードが正しいのにテストが失敗する（誤警報）

どちらもテストスイートへの信頼を損なう。偽陰性はバグの見逃しに直結し、偽陽性は開発者にテスト失敗を無視する習慣を植え付ける。

### 設計フィードバック

テストが書きにくいコードは設計に問題がある。テストの書きにくさは設計の歪みを示すシグナルであり、テストを書くことで設計の改善点が浮き彫りになる。

### 質とスピード

質とスピードはトレードオフではない。内部品質（テスト、設計、リファクタリング）への投資は、中長期的にはスピードの向上として返ってくる。テストを書かないことで得られる短期的なスピードは、技術的負債として蓄積される。

> 出典: t-wada 2024年講演「自動テストの嘘をなくす」
> 出典: t-wada 講演「質とスピード」 https://speakerdeck.com/twada/quality-and-speed

## Kent BeckのTDD 5ステップ

TDDの基本サイクルはKent Beckが定義した以下の5ステップで構成される:

1. **テストを1つ書く** - まず失敗するテストを書く
2. **テストを実行し、失敗を確認する（Red）** - テストが期待通りに失敗することを確認する
3. **テストをパスする最小限のコードを書く（Green）** - 動く最小限の実装を行う
4. **テストを実行し、成功を確認する** - テストが通ることを確認する
5. **リファクタリングする（Refactor）** - 重複を除去し設計を改善する

このRed-Green-Refactorサイクルを繰り返すことで、テストに守られた堅牢なコードを構築する。

> 出典: Kent Beck, "Test-Driven Development: By Example" (2002)

## 原則1: AAAパターン（Arrange-Act-Assert）

**定義**: テストをArrange（準備）- Act（実行）- Assert（検証）の3フェーズに明確に分離するパターンである。Actは1つだけであることが重要である。
**根拠**: フェーズの分離によりテストの意図が明確になる。複数のActを詰め込むとテストの検証対象が不明確になる。
**検出観点**:
- Arrange/Act/Assertの3フェーズが明確に分離されているか
- Actが複数含まれていないか
- フェーズが混在（Act直後にAssert→再びAct）していないか

> 出典: Bill Wake, "3A - Arrange, Act, Assert" (2001) https://xp123.com/articles/3a-arrange-act-assert/

## 原則2: One Assertion Per Test

**定義**: 1つのテストには「意味のあるひとまとまり」のアサーションのみを含めるべきである。1つの振る舞いを検証する複数アサーション（オブジェクトの複数プロパティ検証等）は許容される。
**根拠**: テスト失敗時に何が壊れたかを即座に特定するためである。無関係なアサーションの混在は最初の失敗で後続が実行されず問題の全体像が見えなくなる。
**検出観点**:
- 1テストに無関係な複数のアサーションが詰め込まれていないか
- パラメータ化テストで適切にケースが分離されているか
- 1テスト内で複数の振る舞いを検証していないか

> 出典: Roy Osherove, "The Art of Unit Testing" (2009)

## 原則3: アサーション情報量（Power Assert）

**定義**: テスト失敗時に期待値と実際値が明確に分かるアサーションを書くべきである。t-wada氏はpower-assertの開発者であり、失敗時の情報量を重視する。
**根拠**: デバッガを使わずアサーションメッセージだけで原因を特定できることが理想であり、情報量の少ないアサーションは調査コストを増大させる。
**検出観点**:
- 期待値と実際値が明確なアサーションを使用しているか
- 真偽値のみの検証（`assert result`, `toBeTruthy()`, `t.Fail()`等）はないか
- 例外検証時にメッセージ内容まで検証しているか
- コレクションの検証で要素数や内容まで確認しているか

> 出典: t-wada, "power-assert" https://github.com/power-assert-js/power-assert

## 原則4: 自作自演の回避

**定義**: テストコード内で期待値を計算するロジックが、プロダクションコードと同じロジックを再実装してはならない。
**根拠**: テストとプロダクションコードが同じバグを持つ場合、テストは通るがプログラムは壊れているという最悪の状態になる。期待値はリテラル値で記述するか、別の方法で算出するべきである。
**検出観点**:
- テスト内で期待値をリテラル（ハードコーディング値）で記述しているか
- テスト対象と同じ計算ロジックを使って期待値を算出していないか
- 三角測量（複数の具体例での検証）が適切に使われているか

> 出典: t-wada, "プログラマが知るべき97のこと - テストは正確に、具体的に"

## 原則5: テストのドキュメント性

**定義**: テストはそれ自体がドキュメントとして機能するべきである。テスト名・構造から仕様と意図が読み取れなければならない。
**根拠**: テストは「実行可能な仕様書」であり、失敗時にテスト名だけでどの仕様が壊れたかを判断できることが理想である。
**検出観点**:
- テスト関数/メソッド名が振る舞いを記述しているか
- `test_1`, `test_foo` のような意味のない名前はないか
- テストグループの説明で文脈が説明されているか
- 失敗時にどの仕様が壊れたか分かるか

> 出典: Gerard Meszaros, "xUnit Test Patterns" (2007)

## 原則6: モックの最小化

**定義**: モック（テストダブル）は外部依存の境界にのみ使用し、内部実装のモックは避けるべきである。
**根拠**: 過剰なモックはテストを脆くし、リファクタリングでテストが壊れるようではテストが安全網として機能しない。モックはI/O、ネットワーク、時刻、乱数等の非決定的な外部依存に限定するべきである。
**検出観点**:
- モックは外部依存（I/O、ネットワーク、時刻等）に限定されているか
- 内部実装の詳細（プライベートメソッド等）をモックしていないか
- モックの設定が複雑すぎないか（テスト対象のインターフェースではなく実装に依存していないか）

> 出典: Martin Fowler, "Mocks Aren't Stubs" (2007) https://martinfowler.com/articles/mocksArentStubs.html

## 原則7: テスト信頼性（偽陽性・偽陰性の回避）

**定義**: テストは信頼性が高くなければならない。偽陰性（見逃し）と偽陽性（誤警報）の両方を排除し、テスト結果が常に真実を反映する状態を保つ。
**根拠**: 偽陰性はバグの見逃しに直結し、偽陽性は開発者にテスト失敗を無視する習慣を植え付ける。どちらもテストスイートへの信頼を根本から損なう。

### 偽陰性 (False Negative) の検出観点

- `assert` 文やアサーション関数が1つも無いテスト
- 例外検証のみでメッセージ検証がないテスト
- `pass` だけのテストメソッド
- 例外を握りつぶしているテスト（`try/except` で `pass`）

### 偽陽性 (False Positive) の検出観点

**脆いテスト (Brittle Tests)**:
- 実装の内部詳細に依存するテスト（リファクタリングでテストが壊れる）
- パブリックAPIではなくプライベートメソッドをテストしている
- モックの戻り値が実装の内部構造に依存している

**フレイキーテスト (Flaky Tests)**:
- タイムスタンプ、乱数、実行順序等の非決定的要素に依存
- テスト間の状態共有（グローバル状態、共有データベース等）
- タイムアウトや競合条件に依存するテスト

### コード例（偽陰性）

```python
# 悪い: アサーションがない
def test_process(self) -> None:
    result = process(data)
    # 常にパスする

# 悪い: 例外を握りつぶしている
def test_no_error(self) -> None:
    try:
        risky_operation()
    except Exception:
        pass
```

### コード例（偽陽性 - 言語固有パターン）

**React (実装詳細依存):**
```typescript
// 悪い: getByTestIdの過剰使用
it("shows title", () => {
  render(<Header title="Hello" />);
  expect(screen.getByTestId("header-title")).toHaveTextContent("Hello");
  // getByRoleやgetByTextを使うべき
});
```

**Go (変数キャプチャ漏れ):**
```go
// 悪い: t.Parallel()使用時の変数キャプチャ漏れ（Go 1.21以前）
for _, tt := range tests {
    t.Run(tt.name, func(t *testing.T) {
        t.Parallel()
        // ttがループ変数を参照 - 最後の値でのみテストされる
        got := Add(tt.a, tt.b)
        assert.Equal(t, tt.want, got)
    })
}
```

## 原則8: テストサイズ（テストピラミッド）

**定義**: テストスイートはSmall（単体）テストを土台としたピラミッド型で構成するべきである。Small: 外部依存なし・高速、Medium: ローカルリソース依存・やや低速、Large: ネットワーク/外部サービス依存・低速。
**根拠**: Smallテストが多いほどフィードバックループが短くなりTDDサイクルを高速に回せる。Largeテストは不安定になりやすいため最小限に留める。理想比率はSmall ~80% / Medium ~15% / Large ~5%。
**検出観点**:
- テストの大部分がSmall（外部依存なし、高速）か
- Medium/Largeテストの割合は適切か
- テスト実行時間に問題はないか

> 出典: Mike Cohn, "Succeeding with Agile" (2009)
> Google Testing Blog, "Test Sizes" https://testing.googleblog.com/2010/12/test-sizes.html

## 原則9: テストケースの十分性

**定義**: テストスイートはプロダクションコードのテスト可能な関数・クラスを十分にカバーしているべきである。テスト品質が高くてもテスト対象外の関数が存在すれば信頼性は保証されない。
**根拠**: テストの品質（原則1〜8）が高くてもカバレッジが不十分であればテストスイートの価値は限定的である。特にテストしやすい純粋関数がテストされていない場合は見落としの可能性が高い。
**テスト可能性による優先度**:
1. 純粋関数（最優先） > 2. ユーティリティ関数（高） > 3. ビジネスロジック（高） > 4. I/O依存（中） > 5. エントリポイント（低）
**検出観点**:
- プロダクションコードのpublic関数・クラスに対応するテストが存在するか
- テスト済み関数の主要コードパス（正常系・異常系・分岐）がカバーされているか
- テスト可能性の高い関数（純粋関数、ユーティリティ）が未テストでないか

> 出典: t-wada, "質とスピード" 講演 https://speakerdeck.com/twada/quality-and-speed
> Martin Fowler, "TestCoverage" (2012) https://martinfowler.com/bliki/TestCoverage.html
