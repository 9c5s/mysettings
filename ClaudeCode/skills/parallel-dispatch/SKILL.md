# Parallel Dispatch

複数のスキルをサブエージェントで並列実行し、結果を統合する汎用ディスパッチャー。

Dispatch進捗:
- [ ] Step 1: 入力解析 (スキル・対象の抽出)
- [ ] Step 2: スキル分類 (分析系/実装系)
- [ ] Step 3: 実行計画の提示
- [ ] Step 4: Phase 1 分析系の並列実行
- [ ] Step 5: Phase 2 実装系の実行 (該当時のみ)
- [ ] Step 6: 結果統合
- [ ] Step 7: レポート出力

## 入力フォーマット

`$ARGUMENTS` から以下を抽出する:

- `/xxx` または `/xxx:yyy` 形式 → スキル名
- ファイルパス(`/`で始まらない、拡張子ありまたはディレクトリ) → 対象ファイル
- ブランチ名(`/`で始まらない、拡張子なし、git branch に存在) → ブランチ差分から対象を自動特定
- `--` → スキル名と対象の明示的区切り(省略可)

```
# ファイル指定
/parallel-dispatch /tdd /simplify -- scripts/audio_normalize.py

# ブランチ指定(メインブランチとのdiffから変更ファイルを自動特定)
/parallel-dispatch /tdd /code-review:code-review -- feat/audio-normalize

# 混在(スキル名は/で始まるため自動判別)
/parallel-dispatch /tdd /simplify scripts/audio_normalize.py scripts/tests/test_audio_normalize.py
```

対象が未指定の場合はユーザーに確認する。

## ヘルプ

`$ARGUMENTS` が `help`、`--help`、`-h`、または空の場合、
`references/templates.md` の「ヘルプテキスト」セクションを表示して終了する。

## 実行手順

### Step 1: 入力解析

**ブランチ名が指定された場合:**
メインブランチ名をCLAUDE.mdから取得する(未設定の場合は `main`)。
```bash
git diff <main-branch>...<branch-name> --name-only
```
で変更ファイル一覧を取得し、対象ファイルとする。
ブランチが存在しない場合はエラーを報告する。

### Step 2: スキル分類

各スキルを以下のカテゴリに分類する:

| カテゴリ | 判定基準 | サブエージェント型 | 実行方式 |
|---------|---------|------------------|---------|
| **分析系** | レビュー、チェック、分析、監査等 | `Explore` (読み取り専用) | 並列 |
| **実装系** | 実装、修正、生成、リファクタリング等 | `general-purpose` | 順次 or worktree隔離 |

カスタムスキル(`~/.claude/skills/` 配下)は SKILL.md を Read ツールで読む。
プラグインスキルは description から性質を判断する。
判断に迷う場合は分析系として扱う(安全側に倒す)。

### Step 3: 実行計画の提示

```
実行計画:
- 対象: scripts/audio_normalize.py (+ 1 file from feat/audio-normalize)
- Phase 1 [並列]: /tdd, /simplify, /code-review:code-review (分析系 x 3, Explore)
- Phase 2 [順次]: /test-driven-development (実装系 x 1, general-purpose)
- 実装系スキルにはPhase 1の分析結果をコンテキストとして渡す
```

実装系スキルが複数ある場合:
- 依存関係がある → 依存順に順次実行
- 独立している → worktree隔離で並列実行(`isolation: "worktree"`)
- 判断できない → ユーザーに確認

### Step 4: Phase 1 - 分析系スキルの並列実行

分析系スキルを **全て同一ターンで** バックグラウンド起動する。

**サブエージェント設定:**
- `subagent_type`: `Explore`
- `run_in_background`: `true`
- `description`: `[スキル名] 分析`

プロンプトは `references/templates.md` の「サブエージェントプロンプト」
テンプレートを使用する。

ディスパッチ直後に `references/templates.md` の「進捗テーブル」形式で
進捗を表示し、各エージェント完了時にテーブルを更新する。

### エラーハンドリング

| ケース | 対処 |
|--------|------|
| サブエージェントがタイムアウト | 進捗テーブルで「タイムアウト」と表示し、完了済みの結果のみで統合を進める |
| サブエージェントがエラー終了 | エラー内容をレポートに記録し、残りの結果で統合を続行する |
| Skillツールでスキルが見つからない | サブエージェントプロンプトのフォールバック指示に従う(description推測分析) |
| 分析結果が空(発見事項ゼロ) | 「問題なし」としてレポートに記録する。全スキルが空なら短縮レポートを出力 |
| ブランチが存在しない | エラーメッセージを表示し、正しいブランチ名の確認を促す |
| 対象ファイルが存在しない | 存在しないファイルを報告し、存在するファイルのみで続行するか確認する |

### Step 5: Phase 2 - 実装系スキルの実行(該当する場合のみ)

実装系スキルがある場合、Phase 1の結果をコンテキストとして含めて実行する。

**サブエージェント設定:**
- `subagent_type`: `general-purpose`
- `mode`: `default` (ユーザー承認を求める)
- Phase 1の統合結果を「先行分析の結果」としてプロンプトに含める

複数の実装系スキルの衝突回避はStep 3の方針に従う。

### Step 6: 結果統合

#### 6a. 重複排除
- 同じファイル・行番号への指摘 → 統合
- 同じ根本原因に起因する指摘 → 統合
- 統合元のスキル番号を全て記録する

#### 6b. 優先度決定
- 複数スキルが同一問題を指摘 → 優先度を1段階引き上げ
- 信頼度80未満の単独指摘 → 優先度を1段階引き下げ

#### 6c. 却下判定
以下に該当する指摘は根拠を示して却下する:
- CLAUDE.md やプロジェクト規約に反する指摘
- 明らかに誤った技術的判断
- プロジェクトの意図的な設計判断への不当な干渉

#### 6d. タスク化(分析系のみの場合)
発見事項を実装タスクに変換し、依存関係を特定する。

### Step 7: レポート出力

`references/templates.md` の「レポートテンプレート」の形式で出力する。

## ガイドライン

- 最適なスキル数は **3-5個**。多すぎるとコンテキスト過多で品質が下がる
- 分析系スキルは常に Explore型(読み取り専用)で実行し、コード変更を防ぐ
- 実装系スキルは general-purpose型で実行し、必要に応じて worktree 隔離する
- 結果統合では機械的な結合ではなくプロジェクト文脈を踏まえた判断を加える
- ブランチ指定時はメインブランチ(CLAUDE.mdの設定 or main)との diff を使用する

## 使うべきでない場面

- **単一スキルの実行** → 直接そのスキルを呼ぶ方が効率的
- **スキル間に順序依存がある場合** → 前のスキルの出力が次のスキルの入力になるなら手動で順次実行すべき
- **対象がまだ決まっていない場合** → 先に対象を特定してから実行する
- **探索的な調査段階** → 問題の所在が不明なら、まず単一のExploreエージェントで調査する
